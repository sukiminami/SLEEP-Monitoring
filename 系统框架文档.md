## 系统概述

### 项目简介
本项目是基于 ESP32 平台的睡眠监测系统，通过 R60ABD1 60GHz 毫米波雷达模块采集人体存在、心率、呼吸、睡眠状态等生理数据，并通过多种网络方式（4G/Wi-Fi/蓝牙）将数据以 JSON 格式发送到中国电信天翼物联平台。

### 硬件平台
- **主控芯片**: ESP32-S3
- **雷达模块**: R60ABD1 60GHz 毫米波雷达
- **网络模块**: 
  - A7670C 4G 模块（可选）
  - ESP32 内置 Wi-Fi
  - ESP32 内置 BLE（低功耗蓝牙）

### 软件环境
- **开发框架**: Arduino Framework
- **构建系统**: PlatformIO
- **编程语言**: C/C++

### 协议标准
- **应用层协议**: SM-C0x 睡眠呼吸心跳雷达协议 v1.0（基于中国电信天翼物联平台规范）
- **传输层**: MQTT / TCP
- **数据格式**: JSON

## 核心模块说明

### 1. 雷达模块 (RadarR60A)

**文件位置**: `include/RadarR60A.h`, `src/RadarR60A.cpp`

**功能描述**: 
驱动 R60ABD1 60GHz 毫米波雷达模块，实现人体存在检测、心率监测、呼吸监测、睡眠状态分析等功能。

**核心特性**:
- 串口通信（默认 Serial2，波特率 115200）
- 数据包解析（状态机实现）
- 校验和验证
- 多种数据类型解析（人体存在、呼吸、心率、睡眠、位置等）

**数据结构**:

```cpp
typedef struct {
    bool isDetected;              // 是否检测到人体
    uint8_t breathRate;           // 呼吸频率 (0-35 bpm)
    uint8_t heartRate;            // 心率 (60-120 bpm)
    uint16_t distance;            // 探测距离 (0-65535 cm)
    uint8_t bodyMovement;         // 体动幅度参数 (0-100)
    MovementState movementState;  // 运动状态
    SleepState sleepState;        // 睡眠状态
    BreathStatus breathStatus;    // 呼吸状态
    int16_t positionX;            // X坐标 (cm)
    int16_t positionY;            // Y坐标 (cm)
    int16_t positionZ;            // Z坐标 (cm)
    uint8_t breathWave[5];        // 呼吸波形 (每秒5个值)
    uint8_t heartWave[5];         // 心率波形 (每秒5个值)
    uint16_t awakeDuration;       // 清醒时长 (分钟)
    uint16_t lightSleepDuration;  // 浅睡时长 (分钟)
    uint16_t deepSleepDuration;   // 深睡时长 (分钟)
    uint8_t sleepQuality;         // 睡眠质量评分 (0-100)
    SleepSummary sleepSummary;    // 睡眠综合状态
} RadarData;
```

**通信协议**:
- 帧头: `0x53 0x59`
- 控制字: 标识数据类型
- 命令字: 标识具体命令
- 数据长度: 2 字节
- 数据: 可变长度
- 校验和: 1 字节
- 帧尾: `0x54 0x43`

**控制方法列表**:

### 2. 协议模块 (Protocol)

**文件位置**: `include/Protocol.h`, `src/Protocol.cpp`

**功能描述**:
实现 SM-C0x 睡眠呼吸心跳雷达协议，负责数据帧的构建（上行）和解析（下行）。

**核心特性**:
- JSON 数据帧构建
- 下行指令解析
- 设备标识管理
- 信号强度管理

**数据类型枚举（上行）**:

| 数据类型 | 值 | 描述 |
|----------|----|------|
| `DATA_TYPE_HEARTBEAT` | 0 | 设备心跳信息上报 |
| `DATA_TYPE_PARAMS` | 1 | 设备参数信息上报 |
| `DATA_TYPE_PRESENCE` | 2 | 存在信息上报 |

## SM-C0x 通信协议规范

### 1. 协议概述

SM-C0x 协议是基于中国电信天翼物联平台规范设计的睡眠呼吸心跳雷达通信协议，采用 JSON 格式封装数据帧，支持设备与服务器之间的双向通信。

### 2. 数据帧结构

所有上行数据帧遵循统一的 JSON 格式：

```json

    "imei": "设备IMEI号",
    "imsi": "设备IMSI号",
    "iccid": "SIM卡ICCID号",
    "timestamp": 1234567890,
    "type": 0,
    "signal": 28,
    "sw_version": "1.0",
    "hd_version": "1.0",
    "data": {}
}
```

| 字段 | 类型 | 描述 | 必填 |

## 网络连接方式

### 连接方式对比

| 特性 | 4G (A7670C) | Wi-Fi | BLE (低功耗蓝牙) |
|------|-------------|-------|-----------------|
| **通信距离** | 广域（基站覆盖） | 局域（~100m） | 短距（~10m） |
| **功耗** | 高 | 中 | 极低 |
| **带宽** | 高 | 高 | 中（~1Mbps） |
| **部署成本** | 需要 SIM 卡 | 需要路由器 | 无需额外设备 |
| **适用场景** | 户外/移动场景 | 室内固定场景 | 近距离调试/配置 |
| **初始化时间** | 较长（30-60s） | 短（5-15s） | 短（1-3s） |
| **ESP32-S3 支持** | ✅ 通过串口 | ✅ 内置 | ✅ 内置 |

---

### 网络切换机制

NetworkManager 采用主备网络模式，支持自动故障切换：

1. **主网络优先**: Wi-Fi 作为主网络，4G 作为备用网络
2. **冷却时间**: 两次切换之间至少间隔 10 秒（`SWITCH_COOLDOWN = 10000ms`）
3. **自动切换**: 主网络断开超过冷却时间后自动切换到 4G
4. **自动回切**: 使用 4G 时若 Wi-Fi 恢复，自动切换回 Wi-Fi
5. **定期重试**: 无网络连接时每 60 秒尝试重连 Wi-Fi

## 设计模式

### 1. 策略模式 (Strategy Pattern)

**应用场景**: 网络接口统一管理

**实现方式**:
- 定义抽象基类 `NetworkInterface`，声明所有网络模块必须实现的接口
- 三种网络模块（`A7670C`、`WiFiNetwork`、`BluetoothNetwork`）继承并实现接口
- `NetworkManager` 通过 `_currentNet` 指针指向当前活跃的网络模块
- 运行时可动态切换网络模块，上层代码无需修改

### 2. 回调函数机制 (Callback Pattern)

**应用场景**: 网络状态变化和数据接收通知

**实现方式**:
- 定义回调函数类型：`NetworkStatusCallback` 和 `NetworkDataCallback`
- 回调函数带 `userData` 参数，支持传递上下文数据
- 网络模块在状态变化或接收到数据时触发回调
- 上层代码注册回调函数，实现事件驱动

### 3. 状态机模式 (State Machine Pattern)

**应用场景**: 雷达数据包解析

**实现方式**:
- 定义解析状态枚举：`STATE_IDLE`, `STATE_STX1`, `STATE_CONTROL`, ...
- 根据接收到的字节和当前状态进行状态转移
- 完整接收数据包后触发解析

### 4. 分层架构模式 (Layered Architecture)

**应用场景**: 整体系统架构

**实现方式**:
- 应用层：业务逻辑和调度
- 中间层：协议处理和数据发送
- 驱动层：雷达和网络驱动
- 接口层：网络抽象接口

## 常见问题与解决方案

### 1. 多重定义错误

**问题**: 编译时出现 `multiple definition` 错误

**原因**: 多个源文件中定义了全局变量或函数

**解决方案**: 
- 在 `platformio.ini` 中添加 `build_src_filter` 排除测试文件
- 确保全局变量只在 `.cpp` 文件中定义，在 `.h` 文件中使用 `extern` 声明

### 2. BLE 蓝牙连接问题

**问题**: 手机无法搜索到 BLE 设备或连接失败

**可能原因**:
- BLE 未启用（`USE_BLUETOOTH=0`）
- 设备名称冲突
- 手机 APP 未正确订阅 Notify

**解决方案**:
- 确认 `USE_BLUETOOTH=1` 已启用
- 使用 nRF Connect 或 LightBlue APP 搜索 `SleepMonitor_BLE`
- 连接后订阅 TX 特征的 Notify 功能

### 3. 雷达数据解析失败

**问题**: 无法正确解析雷达数据包

**可能原因**:
- 串口波特率不匹配
- 引脚配置错误
- 校验和计算错误

**解决方案**:
- 确认雷达模块波特率为 115200
- 检查 RX/TX 引脚连接
- 使用调试串口打印原始数据进行对比

### 4. 网络连接超时

**问题**: 4G/Wi-Fi 连接超时

**解决方案**:
- 检查 SIM 卡状态和网络信号
- 确认 Wi-Fi 名称和密码正确
- 增加连接超时时间
- 检查服务器地址和端口是否可达

### 5. 重复类型定义错误

**问题**: 编译时出现 `redefinition` 错误

**原因**: 多个头文件中定义了相同的枚举或结构体

**解决方案**:
- 移除重复定义，保留单一源
- 使用头文件保护宏（`#ifndef ... #define ... #endif`）
- 使用命名空间或前缀区分相同名称的类型
